문제점: 
나는 셀렉트를 커스텀하기 위해 div, ui를 통해서 직접 구현을 했다. 이 셀렉트는 선택시 해당 텍스트가 특정한 p태그 안에 들어가도록 되어 있다.

레거시  코드 원인:  
셀렉트를 제작할 때 선택 전의 상태를 고려하지 않고 제작했다.

가장 좋은 해결책 :
li를 선택시 p태그에 클래스가 추가되도록 구현한다.

가장 좋은 해결책이 안되는 이유:
해당 셀렉트가 컴포넌트가아닌 복붙으로 제작되어 있다.

현실 해결책: 
디자이너를 설득해 선택시 구별하는 것을 빼 버린다. 

얻은점: 
셀렉트를 직접 구현할때 li를 클릭해서 해당 텍스트가 들어가는 곳에 [클래스 추가]가 되도록 만들도록하자.
컴포넌트화를 구체화시키자.

\\\\
커스텀 셀렉트 레거시 코드 유지보수 과정에서, 컴포넌트화되지 않은 복붙 구조로 인해 상태 클래스 추가 방식의 리팩토링이 어려움을 확인함.
전면 수정 대비 영향 범위를 분석한 뒤, 디자이너와 협업하여 선택 상태 시각 표현을 제거하는 방향으로 UX를 조정하여 사이드 이펙트를 최소화함.

2.
문제점:
여러 페이지에서 에디터를 사용중이다. 
페이지 하나의 에디터를 수정하면 다른 페이지의 에디터도 수정되어서 오류가 발생한다.

레거시 코드 원인:
에디터를 제작후 프린트하는 기능을 추가했다. 
문서별로 기본적으로 에디터의 내용이 들어가 있는 경우가 생겼다.
이것을 제작함에 있어서 css가 꼬여서 모든 페이지의 에디터에서 사이드 이펙트가 발생

해결책:
1. 페이지별로 읽는 에디터의 css 파일을 각각만들어 놓자.
- 페이지 별로 읽는 css가 다르기때문에 오류를 방지 할수 있다.
2. 공통 에디터 css는 그대로 두고 섹션/페이지별로 ID를 줘서 에디터 내부의 css를 수정할때는 선택자에 해당 ID를 붙여서 수정한다.
- 하나의 css파일에서 모든 페이지 에디터의 css를 컨트롤 할수 있다.

개인적으로는 2번이 제일 효율적이라고 생각한다.

현실 해결책:
id를 주는 것을 선택
왜냐하면 이미 다양한 css 파일 뒤죽박죽 되어 있기 때문에 이것을 작업하기에는 리소스가 많이 듬
선택자에 id를 추가해서 css 수정하는 방법은 리소스가 줄어들기 때문이다.


\\\\
여러 페이지에서 공통 에디터를 사용하는 구조에서, 프린트 기능 추가로 인해 전역 CSS 충돌이 발생하는 문제를 분석함.
페이지별 CSS 분리안과 ID 스코프 제한 방식을 비교 검토한 후, 공통 CSS는 유지하되 페이지 단위 ID를 통한 선택자 스코프 제한 방식을 적용하여 사이드 이펙트를 해결하고 유지보수성을 개선함.
\\\\

문제사항
프로필 상태값이 휴가·휴직·출장 등 텍스트만 노출되는 구조로,
상태별 색상과 날짜 표시 요구사항이 추가되며 UI 조건 분기가 증가하는 문제 발생

해결 시 고려사항
기존 마크업 구조를 유지하면서 로직 수정만으로 확장이 가능해야 함
상태값이 추가되더라도 조건문이 복잡해지지 않는 구조 필요

해결 방법 및 결과
상태값을 기준으로 UI 속성을 반환하는 computed 로직을 구성하고
v-show 및 클래스 바인딩만 제어하여 마크업 수정 없이 상태별 UI를 구현

원본 데이터: 상태 코드 (숫자 배열)
가공 결과: 상태 텍스트 ,색상 클래스 ,날짜 노출 여부
즉, “상태값 → 화면에 보여줄 UI 정보로 변환”
